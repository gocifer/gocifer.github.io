wp_timer = timer { timeout = wp_timeout }
wp_timer:connect_signal("timeout", function()

for s = 1, screen.count() do
  gears.wallpaper.maximized(wp_path .. wp_files[wp_index], s, true)
end

wp_timer:stop()

wp_index = math.random( 1, #wp_files)
  wp_timer.timeout = wp_timeout
  wp_timer:start()

# 在状态栏右上角添加系统监控的显示。

直接在awesome中使用conky做为系统监控的工具虽好，但conky的浮动状态有些不好控制，控制起来也不像openbox中那么方便，所以还是直接使用awesome自带的系统监控功能算了。我设置了如下的监控内容：

（1）系统内核显示：

先在在rc.lua中添加如下两行代码：

oswidget = wibox.widget.textbox()
vicious.register(oswidget, vicious.widgets.os, " 内核：$2  | " )
1
2

这两行代码，定义了一个显示OS信息的小部件“oswidget”，其中的$2，是显示内核版本，有兴趣可试试从$1~$6，共有6组系统信息。

定义完成，再在rc.lua的后面显示调用处，添加上显示这个小部件的代码，如下：

right_layout:add(oswidget)
1

上述代码，是指定这个小部件显示在状态栏的右边。

（2）CPU状态显示：

与上面的定义方式相似，先定义部件，再加到显示调用的地方，共三行代码，如下：

cpuwidget = wibox.widget.textbox()
vicious.register(cpuwidget, vicious.widgets.cpu, " CPU:$1% | ")
1
2

调用为：

right_layout:add(cpuwidget)
1

（3）内存使用状态显示：

也是与上面定义的方式一样，如下：

memwidget = wibox.widget.textbox()
vicious.register(memwidget, vicious.widgets.mem, " 内存:$1% | ", 20)
1
2

代码尾巴上的20,是显示每20秒刷新一次数据。注意不必设得太快。

调用为：

right_layout:add(memwidget)
1

# 定制标签

只要你愿意，你可以为每一个标签命名，并为每一个标签设置默认布局。这是 rc.lua 里默认的标签设置：

-- { { { Tags
-- Define a tag table which hold all screen tags.
tags = {}
for s = 1, screen.count() do
    -- Each screen has its own tag table.
    tags[s] = awful.tag({ 1, 2, 3, 4, 5, 6, 7, 8, 9 }, s, layouts[1])
end
-- } } }

end

# change transparency

关于终端的透明：

透明效果是通过xcompmgr、transset-df与devilspie这三个小工具实现的。在上述启动项中，已经启动了xcompmgr与“devilspie -a",

在～/用户目录下新建一个文件夹名为“.devilspie”，再在文件夹中新建一个文件名为“opacity.ds”，opacity.ds的内容为：

( if
( contains ( window_class ) "XTerm" )
( begin
( spawn_async (str "transset-df -i " (window_xid) " 0.42" ))
)
)
1
2
3
4
5
6

代码的意思是：如果devilspie捕捉到正在运行的程序的class名叫“XTerm”，那么就用transset-df -i 将这个程序的透明度设为“0.42”。当然，透明的值可根据各人的需要自行设置，0为全透明；1为不透明。 从这段代码不难看出，其实不仅仅是xterm，任何程序都可以在这里为其设置透明度。

举一返三，如果在openbox的autostart文件中执行了”xcompmgr &"与“devilspie -a &”的话，那么在openbox下，也能为各种程序指定透明度。

Some programs use the DPI given by the X server. Examples are i3 (source) and Chromium (source).

To verify that the X Server has properly detected the physical dimensions of your monitor, use the xdpyinfo utility from the xorg-xdpyinfo package:

$ xdpyinfo | grep -B 2 resolution
screen #0:
  dimensions:    3200x1800 pixels (423x238 millimeters)
  resolution:    192x192 dots per inch

This example uses inaccurate dimensions (423mm x 328mm, even though the Dell XPS 9530 has 346mm x 194mm) to have a clean multiple of 96 dpi, in this case 192 dpi. This tends to work better than using the correct DPI — Pango renders fonts crisper in i3 for example.

If the DPI displayed by xdpyinfo is not correct, see Xorg#Display size and DPI for how to fix it.

# Add undetected resolutions

First we run gtf or cvt to get the Modeline for the resolution we want:

$ cvt 1280 1024
# 1280x1024 59.89 Hz (CVT 1.31M4) hsync: 63.67 kHz; pclk: 109.00 MHz
Modeline "1280x1024_60.00"  109.00  1280 1368 1496 1712  1024 1027 1034 1063 -hsync +vsync
1
2
3

Then we create a new xrandr mode. Note that the Modeline keyword needs to be ommited.

$ xrandr --newmode "1280x1024_60.00"  109.00  1280 1368 1496 1712  1024 1027 1034 1063 -hsync +vsync
1

After creating it we need an extra step to add this new mode to our current output (VGA1). We use just the name of the mode, since the parameters have been set previously.

$ xrandr --addmode VGA1 1280x1024_60.00
1

$ xrandr --output VGA1 --mode 1280x1024_60.00Then we create a new xrandr mode. Note that the Modeline keyword needs to be ommited.

# Permanently adding undetected resolutions

Once a suitable resolution is found using xrandr, the mode can be permanently added by creating an entry in /etc/X11/xorg.conf.d/:

/etc/X11/xorg.conf.d/10-monitor.conf
Section "Monitor"
    Identifier "VGA1"
    Modeline "1280x1024_60.00"  109.00  1280 1368 1496 1712  1024 1027 1034 1063 -hsync +vsync
    Option "PreferredMode" "1280x1024_60.00"
EndSection

Section "Screen"
    Identifier "Screen0"
    Monitor "VGA1"
    DefaultDepth 24
    SubSection "Display"
        Modes "1280x1024_60.00"
    EndSubSection
EndSection

Section "Device"
    Identifier "Device0"
    Driver "intel"
EndSection

# 启动设置

首先设置awesome在登录时自动启动。需要设置的地方包括：

  • a、设置“.bash_profile”文件，自动在登录时进入图形界面

我没有安装任何登录管理器。曾经很长时间使用slim做为登录管理器，

后来为精减系统，就养成习惯直接在命令行下运行startx进入图形界面，

再后来，觉得每次登录都手工输入命令还是太麻烦，就直接在.bash_profile中设置为登录时自动进入图形界面。在.bash_profile中添加如下代码即可：

[[ -z $DISPLAY && $XDG_VTNR -eq 1 ]] && exec startx

  • b、设置进入图形界面时，启动awesome窗口管理器。

在用户的.xinitrc文件中，写入：exec awesome 即可。

这里有个小问题————我系统其实有openbox窗口管理器，从.xinitrc文件中启动openbox，命令是：exec openbox-session，

我openbox桌面环境中配置了conky，conky的启动，是从openbox的启动配置文件autostart中启动的。

现在窗口管理器设为awesome ，进入awesome 桌面环境后如果要启动conky，则最好放在现在这个.xinitrc文件中来，不要放到awesome 中去启动，否则conky启动后会报错。原因不明。

也就是说，我现在的.xinitrc文件内容，其实是这样的：

export GTK_IM_MODULE=fcitx
export QT_IM_MODULE=fcitx
export XMODIFIERS=@im=fcitx
exec conky &
##exec openbox-session
exec awesome

# Autostart

To autorun programs, create a shell script via

$ touch ~/.config/awesome/autorun.sh
1

and make it executable by

$ chmod +x ~/.config/awesome/autorun.sh
1

Open autorun.sh in an editor and insert the following:

.config/awesome/autorun.sh

#!/usr/bin/env bash

function run {
  if ! pgrep $1 ;
  then
    $@&
  fi
}

To add programs to autostart, simply append run program [some arguments] to autorun.sh. The run function checks whether there already is an instance of program running and only runs program if there is none. You can check your autorun.sh by running it:

$ ~/.config/awesome/autorun.sh
1

If everything is fine, add the following line to your rc.lua:

.config/awesome/rc.lua

awful.spawn.with_shell("~/.config/awesome/autorun.sh")
